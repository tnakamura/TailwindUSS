# TailwindUSS 実現方法の検討

## 目的

Unity UI Toolkit / USS 上で Tailwind CSS に近い開発体験を提供する Unity 拡張を実現するために、考えられる実現方式をできるだけ幅広く整理する。

この文書では、以下を対象にする。

- Unity Editor 拡張としての提供方法
- Tailwind 風ユーティリティ記法の入力方法
- USS / UXML / C# への変換方法
- 開発体験、保守性、配布性、制約の比較

---

## 先に結論

最初の実装候補としては、次の順で現実的。

1. **ビルド時変換型**  
   UXML / C# / 独自定義ファイルに書かれた Tailwind 風クラスを、Editor 上で USS に変換する方式。
2. **Tailwind CLI 連携型**  
   Node.js の Tailwind CSS をそのまま活用し、Unity 拡張は橋渡しに徹する方式。
3. **C# ネイティブ再実装型**  
   Unity だけで完結するように、Tailwind の一部仕様を C# で再現する方式。

特に MVP としては、**Unity Editor 拡張 + 独自トークン解析 + USS 生成** が最も始めやすい。

---

## 実現方式の一覧

## 1. Unity Editor 内で Tailwind 風クラスを USS に変換する方式

### 概要

- UXML 要素の `class` や独自属性に Tailwind 風クラスを書く
- Unity Editor 拡張がそれを解析する
- 対応する USS ルールを自動生成する
- 生成した USS を UXML に紐づける

### 例のイメージ

- `flex`
- `px-4`
- `py-2`
- `bg-blue-500`
- `text-sm`
- `rounded`

### 長所

- Unity 環境だけで完結しやすい
- Node.js 依存を必須にしなくてよい
- Unity Package として配布しやすい
- Unity 利用者にとって導入が簡単

### 短所

- Tailwind の膨大な仕様を独自実装する必要がある
- バリアントや arbitrary value の再現が難しい
- Tailwind 本家との互換性維持コストが高い

### 向いているケース

- UI Toolkit 向けに機能を絞った Tailwind 風体験を提供したい場合

---

## 2. Tailwind CLI を Unity から呼び出す方式

### 概要

- Unity 拡張はフロントエンドとして振る舞う
- 実際のクラス解決は Tailwind CSS CLI / PostCSS / Node.js に任せる
- 出力された CSS を USS に変換するか、Tailwind の設定情報を使って別途 USS を生成する

### 長所

- Tailwind 本家のエコシステムを活用できる
- `tailwind.config.js` に近い設定運用ができる
- 将来的に theme, plugin, preset への対応余地がある

### 短所

- Node.js の導入が必要
- Unity 利用者には依存関係が重い
- CSS と USS の差分吸収が必要

### 向いているケース

- Web 開発者に馴染みのある Tailwind ワークフローを重視する場合

---

## 3. Tailwind CSS の一部仕様を C# でネイティブ実装する方式

### 概要

- Tailwind の class 名規則を C# で解釈する
- Unity 用のスタイル辞書を持つ
- トークンから USS プロパティ群へ直接マッピングする

### 長所

- 完全に Unity ネイティブで動作する
- Node.js 不要
- パッケージ単体で配布しやすい

### 短所

- 仕様追従コストが大きい
- Tailwind 本家との差異が増えやすい
- CSS エコシステムとの互換性は低い

### 向いているケース

- UI Toolkit に必要なサブセットだけを厳選して提供する場合

---

## 4. 独自 DSL を定義し、それを USS にコンパイルする方式

### 概要

- Tailwind 互換にこだわりすぎず、Unity 向けの独自記法を導入する
- 例えば `.twuxml`, `.twuss`, `.tailwinduss` のような中間ファイルを用意する
- そこから USS と補助 UXML を生成する

### 長所

- Unity / USS の制約に最適化しやすい
- 設計の自由度が高い
- 将来的な拡張がしやすい

### 短所

- Tailwind 既存知識をそのまま使いにくい
- 学習コストが増える
- Tailwind を名乗るには互換性が弱い

### 向いているケース

- Tailwind ライクな思想だけを移植したい場合

---

## 5. UXML の `class` 属性をそのまま Tailwind 互換入力欄として使う方式

### 概要

- UXML 標準の `class` 属性に Tailwind 風クラス列を書く
- 拡張側でそれをスキャンし、対応 USS を作る

### 長所

- 利用者にとって自然
- UXML の既存フローを壊しにくい
- 差分管理しやすい

### 短所

- 本来の USS class と用途が衝突しやすい
- 既存 class 設計との混在ルールが必要

### 補足

- `class="btn flex px-4"` のように、通常クラスとユーティリティクラスを混在させる運用ルールを決める必要がある

---

## 6. UXML に独自属性を追加する方式

### 概要

- 例: `tw="flex px-4 text-sm"`
- 通常の `class` 属性とは分離して保持する

### 長所

- Tailwind 風記法の責務が明確
- 既存 class と競合しにくい
- 独自バリデーションを掛けやすい

### 短所

- UXML 標準仕様との親和性確認が必要
- エディタ拡張やシリアライズの扱いを整理する必要がある

### 向いているケース

- Unity 独自拡張として明快さを重視する場合

---

## 7. C# API で Tailwind 風ユーティリティを指定する方式

### 概要

- UXML ではなく C# から UI を組み立てるケース向け
- `element.AddToClassList("px-4")` などを解析する
- あるいは `Tw.Apply(element, "px-4 py-2 bg-blue-500")` のような API を提供する

### 長所

- C# ベースの UI 構築に対応できる
- エディタ拡張なしでも一部利用しやすい

### 短所

- 静的解析が難しくなる
- ビルド前生成や補完の仕組みが複雑になる

### 向いているケース

- Runtime UI やコード生成 UI を多用する場合

## 27. バリアントを Unity 向けに再定義する方式

### 概要

Tailwind の `hover:`, `focus:`, `dark:`, `md:` などを、Unity に合わせて再定義する。

### 例

- `hover:` → USS 疑似状態相当へ変換できる範囲のみ対応
- `disabled:` → disabled 状態向けルール
- `dark:` → テーマ切替時の別クラス生成
- `sm:`, `md:` → Unity のレスポンシブ概念が弱いため別の意味に置き換える、または非対応

### 長所

- Tailwind 的な思考を残せる

### 短所

- Web と同じ意味にはならない

## 現実的な段階的ロードマップ案

## フェーズ 1: MVP

- Unity Editor 拡張
- UXML の class または独自属性をスキャン
- C# ネイティブ実装で主要ユーティリティのみ対応
- 明示コマンドまたは簡易自動更新で USS 生成
- 対応範囲は layout / spacing / color / typography / border / radius に限定

### 目的

- 最小コストで価値検証する

---

## フェーズ 2: 実用化

- 設定ファイル対応
- 自動再生成
- エラー診断
- 候補補完
- バリアントの限定対応
- C# API 対応

### 目的

- 日常利用できる品質へ上げる

---

## フェーズ 3: 拡張

- ScriptableObject テーマ
- プラグイン拡張ポイント
- UI Builder 連携
- CLI 連携
- Tailwind CLI 連携の検証

### 目的

- エコシステム化と柔軟性向上

---

## おすすめの実現方針

最もバランスがよいのは、以下の組み合わせ。

1. **Tailwind 完全互換ではなく、Unity UI Toolkit 向けサブセットを定義する**
2. **入力は UXML の class 属性または独自 `tw` 属性を採用する**
3. **変換は C# ネイティブで行い、直接 USS を生成する**
4. **生成方式は静的スキャン型 JIT を採用する**
5. **設定は最初は JSON、将来的に ScriptableObject を追加する**
6. **運用は最初は明示コマンド、後から自動再生成を追加する**

この方針なら、Unity への導入性、実装現実性、将来拡張のバランスがよい。

---

## 採用時に最初に決めるべき事項

- Tailwind 互換をどこまで目指すか
- 対応する Unity バージョン
- UI Toolkit 専用かどうか
- 入力媒体を UXML / C# / 独自 DSL のどれにするか
- 出力を単一 USS にするか分割するか
- Node.js 依存を許容するか
- 設定を JSON にするか ScriptableObject にするか
- レスポンシブと arbitrary value を初期対象に含めるか
- クラス名を人間可読にするかハッシュ化するか

---

## まとめ

「Unity で Tailwind CSS を使えるようにする」方法は一つではない。  
最も現実的なのは、**Unity Editor 拡張として Tailwind 風ユーティリティを解析し、UI Toolkit 用の USS を生成する方式**である。

一方で、目指す価値によって適切な方式は変わる。

- **導入性重視**: C# ネイティブ実装
- **互換性重視**: Tailwind CLI 連携
- **Unity 最適化重視**: Tailwind ライクな独自設計

したがって、まずは **Unity 向けサブセット + Editor 生成 + 静的スキャン型 JIT** を軸に始め、その後に設定、補完、バリアント、CLI 連携へ広げるのがよい。
